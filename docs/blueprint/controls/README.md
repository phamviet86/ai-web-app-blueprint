# Machine-readable readiness controls

[`core-controls.json`](core-controls.json) is the closed minimum universe used by guide `08`. It turns the ten scorecard dimensions and four readiness gates into stable IDs without replacing their owning rules. Every core ID is present exactly once; `baseline: true` rows cannot be excluded, while `baseline: false` rows may be `N/A` only when their catalog trigger is absent and the accepted system profile proves that decision.

## Start an assessment

Generate every control and gate so omissions cannot disappear from the denominator:

```bash
python3 docs/blueprint/scripts/score_readiness.py --init docs/readiness/current.json
```

Resolve `assessment_id`, `scorer_version`, `catalog_version`, `system_profile`, `operating_mode`, `source_revision`, `target`, `environment`, `observed_at`, `timezone`, and `freshness_policy` before scoring. `operating_mode` is one of `GREENFIELD`, `EVOLUTION`, `REFACTOR`, `RELEASE`, or `AUDIT`; the first four require their matching gate, while `AUDIT` still requires at least one explicitly selected gate.

For each applicable control, provide boolean `critical`, `critical_rationale`, an allowed numeric `score`, `owner`, `observed_at`, evidence, and either `valid_until` or a non-empty `invalidation_trigger`. Evidence references use `kind:locator`, where kind is `command`, `test`, `artifact`, `measurement`, `drill`, `runtime`, `review`, or `planning`. A score of `0.75` or `1.00` requires at least one observed kind: `command`, `test`, `measurement`, `drill`, or `runtime`. Evidence cannot postdate the assessment; validity cannot end before observation.

For a conditional core control marked `applicable: false`, provide `owner`, `n_a_rationale`, `revisit_trigger`, `decision_observed_at`, and at least one `artifact:` or `review:` `applicability_evidence` locator. Baseline controls cannot be `N/A`. A repo may append a stricter local row with a distinct `CTL-*` ID, `repo_owned: true`, `dimension`, `source_rule`, and `expected_outcome`; selected local controls are scored in that dimension and cannot be added merely to exclude them as `N/A`.

Every applicable gate also records `owner`, `passed`, evidence, `observed_at`, and either `valid_until` or `invalidation_trigger`. A stale passed gate is treated as failed. Non-selected gates use the owned N/A fields generated by `--init`.

## Resolve evidence integrity before readiness

Syntax alone is never readiness proof. Any `0.75`/`1.00` control, passed gate, or conditional N/A decision must resolve its evidence references through the assessment's relative `evidence_manifest` and pinned `evidence_manifest_sha256`. A manifest-backed assessment also names exact `trusted_evidence_producers` and one accountable `evidence_acceptor`. The manifest follows [evidence-manifest.schema.json](evidence-manifest.schema.json) version `1.0.0`, matches the assessment identity/source/target/environment, and maps each reference to an immutable local evidence artifact:

```json
{
  "manifest_id": "EVIDENCE-MANIFEST-001",
  "manifest_version": "1.0.0",
  "assessment_id": "ASSESSMENT-SYSTEM-001",
  "source_revision": "git-sha-or-artifact-id",
  "target": "deployment-or-artifact",
  "environment": "production",
  "records": [{
    "reference": "test:evidence/ci-result.json#contract-suite",
    "artifact": "artifacts/ci-result.json",
    "sha256": "64-lowercase-hex-characters",
    "result": "pass",
    "producer": "ci:protected-contract-test-job",
    "observed_at": "2026-07-12"
  }]
}
```

The scorer verifies manifest/file hashes, identity, dates, passing result, exact producer allowlist membership, evidence acceptor, repository-root path containment, and reference coverage. This proves evidence integrity and detects drift/tampering; it does not independently authenticate the producer or acceptor. The adopting repository must protect its assessment/trust-policy review and name only CI, runtime, drill, or reviewer producers it actually trusts. An unresolved string such as `command:not-executed` can no longer produce a nonzero effective score or `ready`.

Score and print the reproducible calculation:

```bash
python3 docs/blueprint/scripts/score_readiness.py docs/readiness/current.json
python3 docs/blueprint/scripts/score_readiness.py docs/readiness/current.json --json
```

Use an explicit CI/release requirement only where the selected operating mode needs it:

```bash
python3 docs/blueprint/scripts/score_readiness.py docs/readiness/current.json --require ready
python3 docs/blueprint/scripts/score_readiness.py docs/readiness/current.json --require 9.5-ready
python3 docs/blueprint/scripts/score_readiness.py docs/readiness/current.json --expect not-ready
```

Exit code `0` means the assessment is structurally valid and any requested/expected result matched. Exit `1` means a requested readiness level or exact expectation did not pass. Exit `2` means the catalog or assessment is invalid/incomplete. Use `--expect not-ready` for fail-closed fixtures; never use it as a release requirement.

## Catalog lifecycle

- Patch: wording or source-link correction with unchanged control meaning.
- Minor: additive metadata or a new conditional control that does not silently change an existing denominator.
- Major: control removal, split/merge, scoring semantics, gate semantics, or required-field change.
- An assessment pins `catalog_version`; a mismatch is invalid rather than silently rescored.
- Replacing a control requires a migration note in the package changelog and an explicit mapping for active assessments.

The catalog proves that the control universe and formula are reproducible. It does not prove any implementation control; only current assessment evidence can do that.
